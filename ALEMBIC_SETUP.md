# Alembic Setup

Date: 2026-06-09

## Migration Architecture

Alembic is configured inside the backend project:

- `backend/alembic.ini`: Alembic command configuration.
- `backend/migrations/env.py`: Runtime migration environment.
- `backend/migrations/script.py.mako`: Template for future revision files.
- `backend/migrations/versions/0001_initial_schema.py`: Initial migration for the current schema.

The migration environment imports the existing SQLAlchemy `Base` and model modules, then sets `target_metadata = Base.metadata`. This lets Alembic autogenerate future migrations from the current model definitions.

`backend/migrations/env.py` reads the database URL from `app.core.config.settings.DATABASE_URL`, so migration commands use the same configuration source as the FastAPI application.

## How To Create Migrations

Run commands from the `backend` directory.

After changing SQLAlchemy models, generate a new migration:

```bash
alembic revision --autogenerate -m "describe change"
```

Review the generated file in `backend/migrations/versions` before applying it. Autogenerated migrations should be treated as drafts, especially for data migrations, constraint changes, and destructive operations.

## How To Run Upgrades

Run all pending migrations:

```bash
alembic upgrade head
```

Run a specific revision:

```bash
alembic upgrade <revision_id>
```

Use a custom database URL by setting `DATABASE_URL`:

```bash
DATABASE_URL=sqlite:///./local.db alembic upgrade head
```

## How To Run Downgrades

Downgrade one migration:

```bash
alembic downgrade -1
```

Downgrade all migrations back to an empty schema:

```bash
alembic downgrade base
```

## Verification Commands

From the `backend` directory:

```bash
DATABASE_URL=sqlite:///./alembic_verify.db alembic upgrade head
DATABASE_URL=sqlite:///./alembic_verify.db alembic downgrade base
```

Then run tests:

```bash
PYTHONPATH=. DATABASE_URL=sqlite+pysqlite:///:memory: pytest
```

## Assumptions

- Alembic commands are run from the `backend` directory.
- `DATABASE_URL` is available through environment variables or `backend/.env`.
- The initial migration reflects the schema currently declared by the SQLAlchemy models.
- Existing application startup behavior was not changed; `init_db()` still calls `Base.metadata.create_all()` for local starter convenience.
- Future production deployments should rely on Alembic migrations rather than `create_all()`.
